---
title: "Adopt CSS Custom Properties Over Generated Per-Brand CSS"
date: 2020-02-18
context: "GoDaddy's prior styling system generated dozens of static CSS files to cover every brand and localization permutation. Brand updates were slow, risky, and expensive, and the combinatorial output kept growing."
decision: "Move theming to a single runtime layer powered by CSS Custom Properties instead of generating static CSS files per brand and locale."
alternatives:
  - option: "Continue generating static CSS per brand/locale"
    pros:
      - "No new browser requirements"
      - "Fully resolved styles at build time"
    cons:
      - "Combinatorial explosion of files"
      - "Slow, risky brand updates"
      - "Hard to support runtime theming"
  - option: "Compile themes with Sass variables"
    pros:
      - "Familiar tooling"
      - "Build-time validation"
    cons:
      - "Still static, no runtime retheming"
      - "Doesn't reduce the permutation problem"
  - option: "CSS Custom Properties as a runtime theming layer"
    pros:
      - "One stylesheet retheme at runtime"
      - "Eliminates combinatorial file generation"
      - "Now broadly supported across browsers"
    cons:
      - "Requires a baseline of modern browser support"
      - "Runtime, not build-time, resolution"
reasoning: "CSS Custom Properties had become broadly supported, making it possible to retheme a single stylesheet at runtime. This collapsed the file-generation pipeline and made brand, locale, and accessibility variations a matter of swapping values rather than rebuilding stylesheets."
whyItMattered: >-
  The old approach treated every brand and locale as a new build artifact.
  Custom Properties reframed theming as data: define the variables once, swap
  their values per context, and let the cascade do the work. It's the foundation
  everything else at GoDaddy's UXCore was built on.
